## фиксация изменений 

Коммит в Git представляет собой факсацию изменений в вашем проекте. Каждый коммит содержит снимок (snupshot) состояния файлов в определенный момент времени, включая все добавленные, измененные и удаленные файлы. Коммиты составляют историн изменений вашего проекта, позволяя нам отслеживать, кто, когда и какие изменения вносил.

-Для создания коммита используется команда git commit -m "Описание коммита", где "Описание коммита" представляет собой краткое описание внесенных изменений.

-Чтобы просмотреть коммиты и историю изменений, можно использовать команду git log, которая показывает список коммитов в хронологическом порядке.

-Для наглядного представления истории коммитов можно использовать параметр -graph команды git log, который показывает графическое дерево коммитов с ветвлениями и слияниями.

## 2.7 Ветвление

Ветвление (branching) в системе контроля версий Git используется для организации и управления различными линиями разработки. Каждая ветка представляет собой отдельную линию изменений, которую можно развивать независимо от других веток. Ветвление позволяет вам работать над различными фичами, исправлениями ошибок или экспериментами параллельно, не вмешиваясь в основную линию разработки.

## 2.8 Особенности применения "фетч"

"Фетч" (fetch) в контексте системы контроля версий, такой как Git, представляет собой операцию, которая извлекает изменения и обновления из удаленного репозитория, но не объединяет их с вашей локальной веткой. Фетч обновляет информацию о состоянии удаленного репозитория и его ветках в вашем локальном репозитории, но не влияет на вашу текущую рабочую ветку.

## вывод

## 2.13 Исправление коммита

коммита в Git может потребоваться, если вы сделали ошибку в
предыдущем коммите и хотите внести изменения или дополнения в него. Есть несколько способов это сделать, в зависимости от того, какой уровень коррекции вам нужен.

## исправление последнего коммита

![Меню](https://github.com/DanilBorovyagin/software_engine/blob/тема_2/pic/2.13.jpg)

## вывод

-Для исправления последнего коммита используйте git commit --amend, чтобы изменить сообщение коммита или добавить пропущенные файлы.

-Для исправления более раннего коммита используйте git rebase -i с указанием коммита, к которому вы хотите вернуться. Затем замените pick на edit рядом с коммитом, который нужно исправить, и выполните ребейз.

-После редактирования коммита, выполните git commit --amend для сохранения изменений, а затем продолжите ребейз с помощью git rebase --continue.

-Имейте в виду, что изменение коммитов может быть опасным, если другие разработчики уже скачали ваши изменения. В таком случае, необходимо обсудить это с командой и возможно, применить git rebase с опцией --force, чтобы отправить измененные коммиты в удаленный репозиторий.

## 2.15 Настройка. gitignore

.gitignore - это текстовый файл в корневой директории Git-репозитория, который используется для указания Git'y файлов и директорий, которые не должны быть отслеживаемыми системой контроля версий. Это позволяет исключить определенные файлы и директории из репозитория, так что они не будут автоматически добавлены в коммиты и не будут видны другим разработчикам при клонировании репозитория.
